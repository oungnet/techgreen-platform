# แผนปรับปรุงเว็บไซต์และการบูรณาการข้อมูลส่วนกลาง (ฉบับปฏิบัติการ)

อัปเดตล่าสุด: 27 มีนาคม 2026  
โครงการ: TechGreen Platform (Ban Non-Yai Smarter)

## 1) ขอบเขตและวัตถุประสงค์

เอกสารนี้จัดทำเพื่อรองรับการพัฒนาเว็บไซต์ให้พร้อมใช้งานร่วมกับระบบข้อมูลส่วนกลางสำหรับ 3 กลุ่มผู้ใช้งานหลัก:

- หน่วยงานภาครัฐ
- ประชาชนทั่วไป
- ภาคเอกชน/ผู้ประกอบการ

เป้าหมายหลัก:

- ใช้งานข้อมูลเปิดภาครัฐได้ต่อเนื่อง (แม้ API ภายนอกมีข้อจำกัดบางช่วง)
- หน้าเว็บอ่านง่าย ใช้งานง่าย รองรับมือถือ/แท็บเล็ต
- ลดความเสี่ยงลิงก์พัง/ข้อมูลว่าง/ระบบล่มหน้า

## 3) ข้อผิดพลาดที่พบและแก้ไขแล้ว

### 3.1 ปัญหา internal link เป็น `//...`

ผลกระทบ:

- เสี่ยงหลุด base path บน GitHub Pages
- บางลิงก์อาจตีความเป็นโดเมนภายนอกผิด

แนวทางแก้:

- ปรับ `Router base` ให้ไม่สร้าง `//` ซ้ำ
- ยืนยันด้วย browser snapshot แล้วว่า link ภายในกลับมาเป็น `/...` ปกติ

ไฟล์ที่แก้:

- `client/src/main.tsx`

### 3.2 ปัญหา Open Data หน้าโล่งเมื่อข้อมูลภายนอกไม่พร้อม

ผลกระทบ:

- ผู้ใช้เห็นกราฟว่าง/ไม่ทราบสถานะข้อมูล

แนวทางแก้:

- เพิ่มสถานะ `degraded` ให้ชัดเจน
- แสดงเหตุผลที่ข้อมูลหลักไม่พร้อม (ระดับข้อความ)
- แสดงชุดข้อมูลสำรองจากกลุ่มพลังงานต่อเนื่อง
- เพิ่ม empty-state ที่อ่านเข้าใจง่ายแทนกราฟว่าง

ไฟล์ที่แก้:

- `client/src/pages/OpenDataDashboard.tsx`
- `server/services/govDataService.ts`

### 3.3 ปัญหาค่าฐาน URL data.go.th รูปแบบเก่า

ผลกระทบ:

- URL แหล่งข้อมูลอ่านยากหรือซ้ำ endpoint

แนวทางแก้:

- เพิ่มการ normalize ค่า base URL ให้รองรับค่าเก่าได้
- คงพฤติกรรมระบบให้ทำงานต่อได้แม้ env ยังไม่สมบูรณ์

ไฟล์ที่แก้:

- `server/services/govDataService.ts`

## 4) แนวทางออกแบบหน้าเว็บไซต์ (ต่อยอดรอบถัดไป)

### 4.1 โครงหน้าหลักที่แนะนำ

1. หน้าแดชบอร์ดภาพรวม (Landing Dashboard)
- ภาพรวมตัวชี้วัดหลัก
- ปุ่มเข้าใช้งานตามกลุ่มผู้ใช้ (รัฐ/ประชาชน/เอกชน)
- สถานะระบบข้อมูลและประกาศล่าสุด

2. หน้า Open Data สำหรับหน่วยงาน
- ส่วนสถานะ API และคุณภาพข้อมูล
- ตารางข้อมูลตัวอย่างที่ export ได้
- แผง “ความถูกต้องของข้อมูล” (validated/partial/pending)

3. หน้า Open Data สำหรับประชาชน
- ภาษาง่าย อ่านง่าย
- เน้นประโยชน์เชิงปฏิบัติ เช่น ราคาสินค้า แนวโน้มอากาศ ข่าวแจ้งเตือน

4. หน้า Open Data สำหรับภาคเอกชน
- API usage summary
- รายการ dataset ที่เหมาะสำหรับพัฒนาโซลูชันเชิงธุรกิจ

### 4.2 แนวทาง UX ที่ควรยึด

- แสดง `สถานะข้อมูล` ทุกหน้า data-driven (สด/แคช/สำรอง)
- แยก `ข้อมูลไม่พบ` ออกจาก `ระบบผิดพลาด`
- ปรับภาษาบอกข้อผิดพลาดให้เข้าใจง่าย (ไม่ใช้ศัพท์เทคนิคมากเกินไป)
- เพิ่มปุ่มทางลัด “ดูข้อมูลสำรอง” เสมอ

## 5) โครงข้อมูลร่วม (Data Contract ระดับใช้งาน)

สำหรับหน้า Open Data แนะนำให้ยึดโครงมาตรฐาน:

- `status`: `ok | degraded | error`
- `errors`: รายการสาเหตุ (ถ้ามี)
- `fetchedAt`: เวลาที่ดึงล่าสุด
- `cached`: สถานะแคช
- `sourceUrl`: แหล่งข้อมูลต้นทาง
- `preview`: ตัวอย่างข้อมูลสำหรับตรวจสอบด้วยสายตา

ประโยชน์:

- Frontend แสดงสถานะได้สม่ำเสมอ
- ลดกรณีหน้าเว็บว่างโดยไม่มีคำอธิบาย

## 6) ความพร้อมเชิงระบบกลาง (Integration Readiness)

เงื่อนไขขั้นต่ำก่อนเชื่อมระบบกลางจริง:

- Backend online พร้อม health check (`/api/health`)
- ตั้งค่า CORS allow-list สำหรับโดเมน frontend จริง
- ตั้งค่า session cookie ให้เหมาะกับ cross-site (`SameSite=None`, `Secure`)
- มีแผน fallback เมื่อ API ภายนอกไม่พร้อม
